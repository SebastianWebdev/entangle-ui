## What is Entangle UI?

A specialized React component library for **professional editor interfaces** —
3D tools, node editors, IDEs, and agent UIs. The library prioritizes:

- **Density** — compact dimensions, tight spacing, editor-grade hierarchy
- **Dark-first** — every component is designed assuming a dark theme
- **Type-safe theming** — Vanilla Extract zero-runtime CSS, contract-driven
- **React 19+ ESM-only** — no legacy patterns, no `forwardRef`, no UMD
- **Tree-shakable** — every component is `/*#__PURE__*/` annotated

You are working on a library that real developers will pull into production
editor UIs. Quality bar: **every component must look indistinguishable from
hand-written, deeply considered code**. No "AI tells".

## File-by-file structure

Each component lives in its own folder under the right category:

```
ComponentName/
├── ComponentName.tsx           # Main component
├── ComponentName.types.ts      # Props + public types
├── ComponentName.css.ts        # Vanilla Extract styles
├── ComponentName.test.tsx      # Vitest + @testing-library
├── ComponentName.stories.tsx   # Storybook stories (still required for visual checks)
└── index.ts                    # Public exports
```

Categories:

- `src/components/primitives/` — atomic, single-purpose UI atoms
- `src/components/layout/` — components that arrange children in space
- `src/components/form/` — form-related building blocks
- `src/components/navigation/` — menus, tabs, breadcrumbs
- `src/components/feedback/` — alerts, dialogs, toasts, progress
- `src/components/editor/` — editor-specific compositions (PropertyPanel, ChatPanel, etc.)
- `src/components/shell/` — application shell (AppShell, MenuBar, ...)
- `src/hooks/` — exported hooks (each hook in its own subfolder, mirroring `useKeyboard/`)

Each task explicitly tells you which category to use.

## Component authoring pattern (canonical)

````tsx
'use client';

import React from 'react';
import { cx } from '@/utils/cx';
import type { MyComponentProps } from './MyComponent.types';
import { rootRecipe } from './MyComponent.css';

/**
 * Brief one-line summary.
 *
 * Longer description — what problem this solves, when to use it, when not to.
 *
 * @example
 * ```tsx
 * <MyComponent variant="default" size="md">
 *   Content
 * </MyComponent>
 * ```
 */
export const MyComponent = /*#__PURE__*/ React.memo<MyComponentProps>(
  ({
    variant = 'default',
    size = 'md',
    children,
    className,
    style,
    testId,
    ref,
    ...rest
  }) => {
    return (
      <div
        ref={ref}
        className={cx(rootRecipe({ variant, size }), className)}
        style={style}
        data-testid={testId}
        {...rest}
      >
        {children}
      </div>
    );
  }
);

MyComponent.displayName = 'MyComponent';
````

**Hard rules** (deviating costs PR review cycles):

1. React 19 ref-as-prop. **Never** `React.forwardRef`. Always `ref?: React.Ref<HTMLElementType>` in the props interface.
2. `React.memo` with `/*#__PURE__*/` annotation for tree-shaking.
3. `displayName` is required (Storybook docgen depends on it).
4. Public props type is wrapped in `Prettify<...>` from `@/types/utilities`.
5. Props extend `BaseComponent<HTMLElementType>` from `@/types/common`.
6. Use `cx()` from `@/utils/cx` for class merging — never string concat.
7. `@/` path aliases for everything internal — no relative imports crossing folders.
8. `'use client'` directive at the top of every component file.

## Styling pattern (Vanilla Extract)

```ts
import { style, createVar } from '@vanilla-extract/css';
import { recipe } from '@vanilla-extract/recipes';
import { vars } from '@/theme/contract.css';

export const rootRecipe = recipe({
  base: {
    fontFamily: vars.typography.fontFamily.sans,
    color: vars.colors.text.primary,
    transition: `all ${vars.transitions.normal}`,
  },
  variants: {
    size: {
      sm: { height: '20px', padding: `0 ${vars.spacing.sm}` },
      md: { height: '24px', padding: `0 ${vars.spacing.md}` },
      lg: { height: '32px', padding: `0 ${vars.spacing.lg}` },
    },
    variant: {
      default: { background: vars.colors.surface.default },
      filled: { background: vars.colors.accent.primary },
    },
  },
  defaultVariants: { size: 'md', variant: 'default' },
});
```

**Hard rules:**

1. **No hardcoded colors, spacing, radii, transitions, font sizes.** Always `vars.*`.
2. Dynamic runtime values via `createVar()` + `assignInlineVars` from `@vanilla-extract/dynamic`.
3. Use `recipe` for variant combinations; plain `style` for single-purpose styles.
4. When you need state-based styling, use the recipe's variant system — not classnames toggling.
5. Sizes follow the library scale: `Size = 'sm' | 'md' | 'lg'` from `@/types/common`.

## Theme tokens (use these, don't invent)

### Spacing

`xs (2px) · sm (4px) · md (8px) · lg (12px) · xl (16px) · xxl (24px) · xxxl (32px)`

### Font sizes

`xxs (9px) · xs (10px) · sm (11px) · md (12px) · lg (14px) · xl (16px)`

### Border radius

`none (0) · sm (2px) · md (4px) · lg (6px)`

### Transitions

`fast (100ms ease-out) · normal (200ms ease-out) · slow (300ms ease-out)`

### Colors

- `background.{primary, secondary, tertiary, elevated, inset}` — surfaces that own layout
- `surface.{default, hover, active, disabled, whiteOverlay, row, rowHover}` — interactive surfaces
- `border.{default, focus, error, success}`
- `text.{primary, secondary, muted, disabled}`
- `accent.{primary, secondary, success, warning, error}`

(Tokens `background.inset`, `surface.row`, `surface.rowHover` are from v0.7. Use them.)

### Shadows

`sm · md · lg · xl · focus · separatorBottom · separatorRight · separatorLeft · thumb`

### Z-index

`base (1) · dropdown (1000) · popover (1000) · tooltip (1000) · modal (1100)`

## Hooks pattern

For any hook you create or extract:

```
src/hooks/useMyHook/
├── useMyHook.ts           # Implementation
├── useMyHook.test.ts      # Tests
├── useMyHook.types.ts     # Public types (only if non-trivial)
└── index.ts               # Exports
```

Then re-export from `src/hooks/index.ts`.

Hooks must:

- Be SSR-safe — guard `window`, `document`, `ResizeObserver`, etc.
- Clean up — every effect with subscription returns a cleanup function.
- Have stable references — wrap callbacks in `useCallback` with proper deps.
- Be tested with `renderHook` from `@testing-library/react`.

**Important details about the MDX format:**

1. **Filename is kebab-case**: `my-component.mdx`, `segmented-control.mdx`,
   `radio-group.mdx`. Matches the URL slug.

2. **Frontmatter is exactly two fields**: `title` and `description`. No more.

3. **Import paths from MDX are deeply relative** — `../../../../components/...`
   from a page inside `content/docs/components/{category}/`. Don't try to
   shorten with aliases here; Astro's MDX content resolution doesn't follow
   library `@/` aliases.

4. **Every preview must include `client:only="react"`** on the demo component.
   This is non-negotiable — without it, Astro tries to SSR the component and
   the Vanilla Extract runtime fails.

5. **`<PropsTable>`** takes a `props` array of objects with shape
   `{ name, type, default?, description, required? }`. Each prop matches one
   row. Code in `type` and `default` fields is rendered as inline code.
   Required props get a red asterisk via `required: true`.

6. **Markdown tables** (using `|` syntax) are perfectly fine for non-prop
   tabular content (size scales, color mappings, keyboard reference). Use them.

7. **Code blocks** use triple-backtick fenced blocks with `tsx` for usage
   examples. Don't try to make every block live — use real `<ComponentPreview>`
   for live and inert code blocks for tutorial flow.

### Page structure standard (mandatory order)

Every component page follows this structure (skip a section only if not applicable):

1. **Frontmatter**: `title`, `description`
2. **Imports**: `PropsTable`, `ComponentPreview`, demo components
3. **Intro paragraph**: 1-3 sentences. What problem does it solve? When to reach for it?
4. **Live Preview**: `<ComponentPreview title="Live Preview">` with default demo
5. **Import**: code block showing the import statement
6. **Usage**: code block(s) showing common usage
7. **Variants / Sizes / States / etc.**: each with a `<ComponentPreview>` + code block + optional table
8. **Composition / Recipes**: for compound components (e.g. `Alert.Title` / `Alert.Description`) or common patterns
9. **Accessibility**: bullet list of keyboard interactions, ARIA semantics, focus behavior
10. **API Reference**: `<PropsTable>` for each public component (parent + sub-components separately if compound)

### Writing style

- Concise, opinionated, no marketing fluff
- Direct sentences. No "Welcome to the Button component!" — start with substance
- "When to use" / "When not to use" earn their own short paragraph if the
  distinction is non-obvious (e.g. Alert vs Toast, Radio vs SegmentedControl)
- Address the reader directly ("you") — not formally ("the user", "developers")

## Branch & PR workflow

Every task in this folder corresponds to **one branch** and **one PR**. The
branch is named in the task header. **Never work directly on `main`** — if
you find yourself on `main`, create the task's branch before making any
changes.

### Branch naming

Each task file specifies its branch name in the header (e.g. `feat/radio-component`).
Use that name verbatim. Do not invent variants.

The repo's existing convention for ticketed work is `etui-<ticket>-<task-name>`.
The v0.8 release plan uses semantic prefixes (`feat/`, `chore/`) since these
tasks aren't tied to an external ticketing system. When a ticket exists,
prefer `etui-<ticket>-<task-name>` over the semantic prefix and update the
task header accordingly.

### Base branch

> **Read this if you are an agent picking up a v0.8 task.**
>
> Every v0.8 task branch is **created from `release/v0.8.0`** and **merged
> back into `release/v0.8.0`** — NOT `main`. `release/v0.8.0` already exists
> on origin; do not recreate it from `main`.
>
> Branching from `main` instead of `release/v0.8.0` is the most common
> mistake on this release. It produces PRs that miss earlier v0.8 work,
> conflict on `package.json` / lock files, and break the conventions doc
> chain. **If you find yourself on `main`, stop and switch.**

The correct opening sequence for a v0.8 task is:

```bash
git fetch origin
git checkout release/v0.8.0
git pull --ff-only origin release/v0.8.0
git checkout -b <branch-name-from-task-header>
```

`<branch-name-from-task-header>` is taken verbatim from the task file
(e.g. `feat/skeleton-component`). The PR for that branch targets
`release/v0.8.0` as its base.

#### Why this matters

- **`release/v0.8.0` is the integration branch** for the entire v0.8
  release. All 19 tasks land there, accumulate their changesets, and ship
  to `main` (and to npm) as a single coordinated release.
- **`main` only updates at release time**, when `release/v0.8.0` is merged
  in as a whole. Until then, `main` reflects the last shipped version
  (currently 0.7.x).
- **`docs/0.8-conventions.md` lives on `release/v0.8.0`**, not `main`.
  Branching from `main` means starting without this guide.
- **Earlier merged tasks live on `release/v0.8.0`**. Branching from `main`
  means re-fighting conflicts that were already resolved upstream.

#### When `release/v0.8.0` did not exist yet

The first task in the release created `release/v0.8.0` from the latest
`main` and pushed it. That step is already done — every subsequent task
branches from the integration branch. If you are starting v0.9 work,
repeat that bootstrap on `release/v0.9.0`.

## Release workflow: Changesets

The repo uses [Changesets](https://github.com/changesets/changesets) for
versioning and changelog generation. **Every PR that ships a user-visible
change must include a changeset file.** This is non-negotiable — the release
process aggregates changesets to produce the CHANGELOG and the next version
number.

The relevant config:

- `.changeset/config.json` uses `@changesets/changelog-github` with the
  GitHub repo `SebastianWebdev/entangle-ui`
- Base branch is `main`
- No auto-commit (changesets are committed manually as part of the PR)
- Public access on npm
- The release workflow uses npm OIDC Trusted Publishing

### When to create a changeset

| Type of change                                                          | Changeset?              | Bump type                                      |
| ----------------------------------------------------------------------- | ----------------------- | ---------------------------------------------- |
| New component / hook (T-01 through T-16)                                | yes                     | **minor**                                      |
| New theme preset (T-14)                                                 | yes                     | **minor**                                      |
| New utility export (T-07's `platform.ts`, T-18's tokens.json artifacts) | yes                     | **minor**                                      |
| Bug fix in existing component                                           | yes                     | **patch**                                      |
| Refactor with no API change (T-12 hook extractions, internal cleanup)   | yes                     | **patch**                                      |
| Breaking API change                                                     | yes                     | **major** (don't ship in v0.8 — defer to v1.0) |
| Pure docs change with no source changes                                 | optional — usually skip |
| CI / build config changes only                                          | usually skip            |
| Storybook stories only (no source changes)                              | skip                    |

When in doubt, **create a changeset**. Empty PRs without changesets get
ignored by the release process and won't appear in the CHANGELOG.

### How to create a changeset

After implementing the task and before committing:

```bash
npx changeset
```

Walk through the interactive prompt:

1. Pick the bump type (`minor` for new features, `patch` for fixes)
2. Write a **user-facing summary** — one or two sentences describing what
   shipped from the consumer's perspective. This text becomes the changelog
   entry, so write it for someone reading release notes, not for someone
   reviewing the PR.

Examples of good changeset summaries:

```
Add Radio and RadioGroup primitives for mutually-exclusive form selections.
Closes the last Phase-1 gap.
```

```
Add Skeleton and SkeletonGroup components for loading placeholder states.
Supports rect / circle / line shapes with pulse, wave, or no animation.
```

```
Add useDisclosure, useClipboard, useClickOutside, and useHotkey hooks to
the public API.
```

Examples of bad changeset summaries:

```
implement task T-01           ← Internal, meaningless to consumers
add Radio.tsx and tests       ← Implementation detail, not user-facing
PR review fixes               ← Review feedback isn't a feature
```

The changeset CLI writes a markdown file like `.changeset/curly-foxes-dance.md`
with the bump type in frontmatter and the summary as body. Commit this file
alongside the rest of the PR.

### Aggregation: how changesets become a release PR

Once T-01's PR merges to `release/v0.8.0`, the changeset file lives in
`.changeset/`. T-02's PR adds another changeset file. By the time all 19
tasks merge, `release/v0.8.0` contains 19 changeset files.

When the release manager opens a PR from `release/v0.8.0` to `main`:

1. The Changesets GitHub Action runs `changeset version`
2. It consumes all `.changeset/*.md` files
3. Bumps `package.json` to the next version (highest bump type wins —
   one `minor` makes the whole release `minor`)
4. Generates a unified `CHANGELOG.md` entry with sections for each bump type
5. Deletes the consumed changeset files
6. Commits the version bump + changelog to the release PR

When that release PR merges to `main`, the publish workflow:

1. Runs `npm run release` → `changeset publish`
2. Publishes the new version to npm (via OIDC Trusted Publishing — no
   long-lived tokens)
3. Tags the commit with `v0.8.0`
4. Creates a GitHub Release with the changelog body

**The agent's only responsibility**: create the changeset file in their
PR. Aggregation is automatic.

### Multiple commits in one PR

A PR may have multiple commits but **one changeset file**. The changeset
covers the whole PR's user-visible change, not individual commits. If a
PR is so big it would need multiple changesets, that's a sign it should be
split into multiple PRs.

### Forgot to add a changeset?

If a PR is opened without a changeset, the bot will comment requesting
one. Don't merge without it. Add the changeset in a follow-up commit on
the same branch.

## Pre-commit checklist (run BEFORE every commit)

Run these in order. Each step blocks the next if it fails.

```bash
1. npm run lint           # Fix any errors before continuing
2. npm run type-check     # Ensure no TypeScript errors
3. npm run format         # Format all changed files

## Coding-agent etiquette

When implementing a task:

1. **Read this file first.** Then read the task file. Then start.
2. **Read existing similar components first.** Want to write Skeleton? Look at
   how Spinner (from v0.7) or Switch is written. Want to write a hook? Look at
   `useKeyboard`. The patterns are already established.
3. **Don't invent design tokens.** If you need a new spacing/color value, that's
   a flag — bring it up rather than hardcoding.
4. **Don't break consistency.** If existing components use a certain prop name
   pattern (e.g. `size: 'sm' | 'md' | 'lg'`), match it. Don't introduce
   `compact: boolean`.
5. **Stay in scope.** A task that says "add Radio" doesn't mean "and also
   refactor Checkbox to share code". Scope creep blocks releases.
6. **One task per PR.** Each task in `v0.8/` corresponds to one branch and one PR.
